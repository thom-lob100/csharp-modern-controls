# 현업 담당자용 — Codex로 기존 화면을 모던 컨트롤로 바꾸는 방법

이 문서는 **개발 지식이 깊지 않아도** Codex(사내 AI 코딩 도구)에게 지시해서
기존 프로그램 화면을 모던 컨트롤로 바꿀 수 있도록 만든 사용 설명서입니다.
아래 프롬프트 상자를 **그대로 복사–붙여넣기** 하면서 진행하면 됩니다.

---

## 0. 시작 전 준비 (최초 1회)

> **IT가 세팅을 다 해주는 경우 (권장)**: 아래 체크리스트가 전부 완료된 PC를
> 받았다면 이 장은 건너뛰고 **1장부터** 시작하면 됩니다.
>
> **[IT 세팅 체크리스트]**
> - [ ] 개발환경(Visual Studio) + 소스 + 모던 컨트롤 라이브러리/`docs` 폴더 설치
> - [ ] 회사 솔루션에 `Modern.Lab.Commons` 프로젝트/참조 + WPF 어셈블리 참조 연결
> - [ ] 솔루션 루트에 `AGENTS.md` 복사 + `{{ }}` 값 전부 채움 (빌드 명령 포함)
> - [ ] 손대지 않은 상태로 **기준선 빌드 성공** 확인
> - [ ] 원본 전체를 zip 등으로 1회 백업해 별도 보관
>       (git/Bitbucket을 쓸 수 있는 환경이면 저장소 초기화 + 원본 커밋으로 대체)
> - [ ] Codex가 명령 실행(빌드, 파일 복사)을 할 수 있는 권한 모드로 설정
> - [ ] (스프레드 화면을 변환할 계획이면) `ModernSpreadGrid` 최초 1회 연결 완료
>       — 절차는 `docs/migration/ModernSpreadGrid.md`
>
> 아래 내용은 세팅이 안 된 PC에서 Codex에게 직접 세팅을 시킬 때만 사용합니다.

**사람이 해둘 것:**

- [ ] 회사 프로그램 소스 폴더(예: `C:\smartAos`) 아래에 모던 컨트롤 라이브러리
      폴더(`Modern.Lab.Commons` 소스 + `docs` 문서 폴더 전체)를 복사해 두었다
      (예: `C:\smartAos\Libs\ModernLab\`).
- [ ] 변환한 프로그램을 **직접 실행해서 확인하는 방법**을 알고 있다
      (어떤 exe를 켜고, 어떤 메뉴에서 그 화면을 여는지).

**나머지 설정은 Codex에게 시킵니다.** 소스 폴더에서 Codex를 열고 아래를
붙여넣으세요 (최초 1회):

```
이 폴더는 회사 WinForms 프로그램 소스이고, 기존 화면을 모던 컨트롤로 바꾸는
작업을 준비하려고 한다. 아래를 순서대로 수행하고 각 항목의 결과를 보고해라.

1. 수정 전 원본을 보존해라: 이 폴더 전체를 bin/obj 등 빌드 산출물은 빼고
   _backup\원본_<오늘날짜>.zip 으로 압축 백업해라. 이미 있으면 생략해라.
   (이 환경은 git을 쓰지 않는다. git 명령은 사용하지 마라.)
2. 모던 컨트롤 라이브러리 폴더(Modern.Lab.Commons가 들어 있는 폴더)를 찾고,
   docs/codex/AGENTS.md 템플릿을 이 폴더 루트에 AGENTS.md로 복사해라.
3. AGENTS.md의 {{ }} 자리표시자를 전부 실제 값으로 채워라:
   - 회사 솔루션(.sln) 경로를 찾아 넣어라. 여러 개면 목록을 보여주고 멈춰라.
   - vswhere 또는 설치 경로 탐색으로 MSBuild.exe 전체 경로를 찾아
     빌드 명령을 완성해라.
   - 라이브러리 경로와 docs 폴더 경로를 넣어라.
   - 수정 금지 폴더(Java 서버 소스 등)가 있으면 그 경로를 넣어라.
4. 회사 솔루션이 Modern.Lab.Commons를 참조하게 만들어라:
   솔루션에 Modern.Lab.Commons 프로젝트를 추가하고, 화면(폼)이 있는 프로젝트에
   프로젝트 참조와 WindowsFormsIntegration, PresentationCore,
   PresentationFramework, WindowsBase, System.Xaml 참조를 추가해라.
   docs/codex/form-conversion-guide.md 0장의 체크리스트를 기준으로 삼아라.
   이미 참조가 되어 있으면 확인 결과만 보고해라.
5. 아무 폼도 바꾸지 않은 현재 상태로 솔루션을 빌드해서 성공하는지 확인해라.
   이것이 기준선이다. 실패하면 원인만 보고하고 임의로 고치지 마라.
6. 이번에 만들거나 수정한 파일 목록을 정리해서 보고해라.
```

보고에서 **5번(기준선 빌드)이 성공**인지 꼭 확인하세요. 실패라면 그 보고서를
IT 담당자에게 전달하고, 성공한 뒤에 1장으로 진행합니다.

## 1. 꼭 지켜야 할 세 가지

1. **한 번에 화면(폼) 하나씩만** 바꿉니다. "전부 다 바꿔줘"라고 하지 마세요.
2. 단계마다 Codex가 **멈추고 보고**하게 되어 있습니다. 보고를 읽고 이상 없을 때만
   다음 프롬프트를 주세요.
3. 화면 바꾸는 작업에 **기능 개선 요청을 섞지 마세요** ("바꾸는 김에 정렬 기능도
   추가해줘" 금지). 동작이 바뀌면 어디서 바뀌었는지 찾기 어려워집니다.

## 2. 작업 순서 — 프롬프트 복사해서 사용

### ① 안전지점 만들기 (되돌리기 준비)

```
지금부터 <폼이름> 폼을 모던 컨트롤로 변환하는 작업을 시작한다.
작업 전에 안전지점을 만들어라:
이 폼과 관련된 파일(<폼이름>.cs, <폼이름>.Designer.cs, <폼이름>.resx 등)을
전부 찾아 _backup\<오늘날짜>_<폼이름>\ 폴더에 복사하고,
복사한 파일 목록을 보고해라.
이 환경은 git을 쓰지 않는다. 아직 다른 작업은 하지 마라.
```

`<폼이름>` 자리에 실제 폼 이름을 넣으세요 (예: `EmployeeListForm`).
폼 이름을 모르면 먼저 이렇게 물어보세요:

```
이 솔루션에서 "<화면에 보이는 프로그램/메뉴 이름>" 화면에 해당하는
폼 파일이 무엇인지 찾아서 알려줘. 아직 아무것도 수정하지 마라.
```

### ② 조사 시키기 (아직 코드를 바꾸지 않음)

```
AGENTS.md와 docs/codex/form-conversion-guide.md 절차서를 따라
<폼이름> 폼의 3-1단계(조사와 계획)만 수행해라.
컨트롤 인벤토리 표, 보류 항목, 예상 코드 보정 목록을 보고하고 멈춰라.
코드는 아직 한 줄도 수정하지 마라.
```

**보고서에서 볼 것:**
- "보류 항목"이 있으면 → 그 컨트롤은 이번에 안 바뀝니다. 화면의 핵심
  컨트롤(예: 메인 그리드)이 보류라면 이 화면은 변환을 미루고 IT와 상의하세요.
- 표가 이해되지 않아도 괜찮습니다. "보류 없음 / 몇 개 교체"만 확인하면 됩니다.

### ③ 변환 실행

```
좋다. 방금 보고한 계획대로 3-2 ~ 3-6단계를 수행해라.
절차서의 절대 규칙(업무 코드 수정 금지, 한 폼만, 빌드 확인)을 지키고,
끝나면 AGENTS.md의 완료 보고 형식으로 보고해라.
```

**보고서에서 볼 것:**
- `빌드: 성공` 이어야 합니다. 실패라면 ④로.
- "보류 항목"과 "실행 확인 필요 항목"을 메모해 두세요 (⑤에서 사용).

### ④ (필요할 때만) 빌드 오류 수정

```
빌드 오류를 절차서 3-5단계에 따라 수정해라.
업무 로직(서버 호출, 저장, 계산)은 절대 건드리지 말고,
어떤 파일의 어떤 줄을 왜 고쳤는지 보고해라.
```

같은 오류로 3번 넘게 반복되면 중단하고 IT 담당자에게 보고서를 전달하세요.

### ⑤ 사람이 직접 확인 (제일 중요한 단계)

프로그램을 **직접 실행**해서 아래를 확인합니다:

1. 변환한 화면이 열린다 (오류 창 없이).
2. Codex 보고서의 "실행 확인 필요 항목"을 **하나씩 전부** 해본다
   (조회 버튼, 콤보 선택, 행 선택, 저장 등).
3. 변환 전과 **동작이 같은지**를 봅니다. 모양은 달라져도 되고(그게 목적),
   동작·데이터가 다르면 안 됩니다.

이상이 있으면 그 내용을 그대로 Codex에게 알려주세요:

```
실행해 보니 문제가 있다: <본 것을 그대로 서술 —
예: "조회 버튼을 누르면 예전에는 그리드에 20건이 나왔는데 지금은 비어 있다">
원인을 조사해서 보고부터 해라. 바로 고치지 말고 원인과 수정 계획을 먼저 보여줘.
```

### ⑥ 마무리 (확인이 끝났을 때)

```
확인 완료했다. 이번 변환에서 바뀐 파일 전체 목록과 파일별 변경 요약을
보고해라. _backup\<오늘날짜>_<폼이름>\ 폴더는 지우지 말고 그대로 둬라.
```

백업 폴더는 다음 폼 몇 개를 더 변환하고 프로그램이 안정된 뒤에 지우세요.
형상관리(Bitbucket) 반영은 회사 절차에 따라 IT 담당자에게 요청하세요.

## 3. 되돌리고 싶을 때

확인 결과가 이상하고 원인 파악도 어렵다면, 언제든 안전지점으로 돌아갈 수 있습니다:

```
이번 변환 작업을 전부 취소한다.
_backup\<오늘날짜>_<폼이름>\ 폴더의 파일들을 각자 원래 위치에 덮어써서
복원하고, 복원 후 솔루션 빌드가 성공하는지 확인해서 보고해라.
```

①에서 백업을 만들어 두었기 때문에 원래 소스는 손상되지 않습니다.
(백업 폴더까지 잃어버린 경우에는 IT가 보관 중인 원본 전체 zip에서 복원합니다.)

## 4. 자주 묻는 상황

| 상황 | 대처 |
|---|---|
| Codex가 계획 없이 바로 코드를 고치기 시작한다 | "멈춰라. 절차서 3-1단계 조사 보고부터 해라"라고 입력 |
| 보고서에 모르는 용어가 많다 | "이 보고서를 비개발자가 이해할 수 있게 다시 설명해줘"라고 요청 |
| "이 컨트롤은 대응하는 모던 컨트롤이 없다"고 한다 | 정상입니다. 그 컨트롤만 기존 모습으로 남습니다. 보고서에 기록됐는지만 확인 |
| 화면 색/글꼴이 예전과 다르다 | 정상입니다(모던 테마가 외형을 통일). 동작만 같으면 됩니다 |
| 여러 화면을 빨리 바꾸고 싶다 | 그래도 한 화면씩. ①~⑥이 익숙해지면 한 화면에 10~20분이면 끝납니다 |
| 작업 중 Codex 세션이 끊겼다 | 새 세션에서 "_backup\<날짜>_<폼이름> 폴더의 원본과 현재 소스를 비교해서, 절차서 기준으로 어디까지 진행됐는지 보고해라" |

## 5. IT 담당자에게 넘길 것 (문제 발생 시)

- Codex의 마지막 보고서 전체 복사본
- 실행해서 본 증상 (스크린샷이 있으면 더 좋음)
- 백업 폴더 이름 (`_backup\<날짜>_<폼이름>`)
